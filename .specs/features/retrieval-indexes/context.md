# Context — retrieval-indexes (Cycle 5, TDD Phase 6)

Decisions captured during Specify/Design. Run inside `learny-ship-cycle` under the
**auto-decision rule**: each gray area is resolved with an explicit option set
(why-recommend AND why-not), the recommended option chosen, and recorded here +
as an `AD-NNN` row in `.specs/STATE.md`. None met the ship-cycle escalation bar
(no provider is locked in — see D-1; no MVP-scope change) so none were escalated.
These are inputs to `design.md` and are not re-opened there.

| ID | Gray area | Options (✅ = chosen) | Rationale (why-recommend / why-not) |
|---|---|---|---|
| D-1 | **Embedding provider** — wire a concrete cloud model now, or defer? | ✅ **(a)** Ship a Learny `EmbeddingPort` (domain) + a **deterministic, dependency-free local adapter** as the default; defer the production cloud provider/model to its own ADR when Phase 7 needs real semantic quality. **(b)** Wire OpenAI embeddings now. | (a) keeps retrieval fully testable and provider-agnostic, honors CLAUDE.md ("do not assume a concrete provider/model default until a follow-up ADR") and ADR-0007 (provider behind a port), and adds no SDK/API-key requirement to run the worker or tests. Cost: the semantic arm is not *semantically* meaningful until a real model is wired — mitigated because the lexical (FTS) arm carries real matching and the full vector path (column, HNSW, `register_vector`, cosine, RRF fusion) is exercised end-to-end. (b) gives real semantic quality immediately but locks in a provider CLAUDE.md explicitly defers, makes the worker/tests depend on a network + secret, and pre-empts the follow-up ADR. |
| D-2 | **Embedding storage shape** | ✅ **(a)** Nullable `embedding vector(1536)` column on `corpus_chunks`. **(b)** Separate `corpus_chunk_embeddings` 1:1 table. | (a) is simplest, matches the `pgvector-hybrid-search` skill, needs no extra join in the retrieval query, and is nullable because embeddings are populated asynchronously after corpus build (HNSW skips NULLs). Cost: mixes a derived vector onto the canonical chunk row; a dimension change is an `ALTER`. (b) isolates the derived vector but adds a join to every retrieval and more schema for no MVP-scale benefit. |
| D-3 | **Vector type / distance / index** | ✅ `vector(1536)`, cosine (`vector_cosine_ops`, `<=>`), **HNSW** `WITH (m=16, ef_construction=64)`. | 1536 is a widely-used, provider-neutral embedding size (fits common + truncatable models); cosine is the default for normalized embeddings; HNSW needs no training step so it builds on the (initially empty) table and survives re-indexing. Cost: a future model with a different dimension needs a dim-change migration + re-embed — accepted, because embeddings are derived and re-indexable (ADR-0001). `LEARNY_EMBEDDING_DIM` is recorded as informational; the migration column dim is the literal source of truth. |
| D-4 | **Lexical / full-text field** | ✅ A `STORED` generated `search_vector tsvector` on `corpus_chunks`: `setweight(to_tsvector('english', <deepest section title>), 'A') \|\| setweight(to_tsvector('english', coalesce(text,'')), 'D')`, GIN-indexed; `websearch_to_tsquery('english', :q)` at query time, `ts_rank_cd(..., 32)` scoring. Section title = `section_path ->> -1` (guarded). | Skill-prescribed; title-weighted-over-body matches citation-sensitive lookup; a generated STORED column stays correct on every write with zero external call and is computed for pre-existing rows at `ALTER` time. Cost: single `'english'` regconfig — non-English books rank worse; accepted for MVP, multi-language `regconfig` is a future concern. |
| D-5 | **Where embedding runs in the pipeline** | ✅ **(a)** The `run_ingestion` Celery task runs an **embed+index step after** corpus build, in its **own committed transaction**, before `complete`, reusing the existing retry/terminal classification; an `EmbedCorpus` application service reads the source's chunks and `UPDATE`s their vectors in batches via `EmbeddingPort.embed_documents`. **(b)** A separate chained `embed.run` Celery task. | (a) mirrors the established "each step in its own committed unit of work" pattern (`tasks.py`), keeps the network call **out of** the corpus-write transaction, is idempotent/re-runnable, and needs no second queue. Cost: embedding a large book is one long step — accepted (batched + retryable). (b) adds task/queue/retry machinery for no MVP benefit; pipeline-as-separate-tasks is a future optimization. |
| D-6 | **Retrieval query + evidence** | ✅ Single-statement hybrid **RRF** query (two ranked CTEs per arm, `FULL OUTER JOIN`, `Σ 1/(k+rank)`) over `corpus_chunks`, **scoped to one `source_id`**, projecting citation anchors into a frozen `Evidence` domain result (`chunk_id, source_id, section_path, anchor, page_span, snippet, score`); the semantic arm excludes NULL-embedding chunks. A `RetrievalPort` fronts the SQL. Tuning (`semantic_limit`, `lexical_limit`, `rrf_k=60`, `top_k`, `hnsw_ef_search`) lives in `LEARNY_` settings. | Skill-prescribed; one testable seam for golden fixtures; anchors make every result independently citable (ADR-0003). Retrieval is **per-source** because Phase 7 Q&A and Phase 8 teaching both target one book. No reranker (deferred by ADR-0006). |
| D-7 | **Vertical-slice scope** (most likely to surprise — flagged at the merge gate) | ✅ **(a)** Backend capability **+ a thin owner-scoped `POST /api/sources/{id}/retrieve` endpoint** returning evidence; **no frontend** this cycle. **(b)** Backend capability only, tested via golden fixtures, no endpoint. **(c)** Full slice incl. Next.js proxy + a search-results UI. | (a) makes the phase independently demoable/testable against a real route and gives Phase 7 the exact seam it consumes, while **not** building throwaway raw-search UI — raw hybrid results are not an MVP user surface (the MVP surfaces are cited Q&A and teaching, Phases 7–8). It partially departs from AD-010's full-vertical-slice cadence, but AD-010 says "matching Cycle 1 cadence" and Cycle 3 already shipped a backend-heavy slice; retrieval has no natural standalone user screen, and the endpoint is retained (Phase 7 reuses it + the empty-result path drives "not found in source"). (b) is not demoable in the running app. (c) builds retrieval-debug UI outside MVP product scope. |
| D-8 | **Endpoint input / output contract** | ✅ Body `{ "query": str, "top_k"?: int }`; empty/whitespace `query` → 422; `top_k` bounded `1..LEARNY_RETRIEVAL_MAX_TOP_K` (default from settings); missing/non-owned source → **404** (mirrors the structure endpoint's ownership-as-404); **no** results → **200** with an empty evidence list (the surface Phase 7 turns into "not found in source"). No new rate limit (read-shaped; consistent with the structure endpoint). | Consistent with existing owner-scoped read endpoints; the empty-result 200 is deliberate — it is the Phase-7 hook, not an error. |
| D-9 | **Test fixtures / determinism** | ✅ Golden-fixture retrieval tests build a corpus from the **existing synthetic EPUB fixtures**, embed via the deterministic local adapter (reproducible vectors), and assert expected `chunk_id`s/anchors appear in the candidate set for fixture queries (Phase-6 success metric). Integration tests need the `vector` extension in the test DB. | Reuses Cycle-4 fixtures; deterministic embeddings make the semantic arm reproducible without a network; the FTS arm asserts real lexical recall. |
