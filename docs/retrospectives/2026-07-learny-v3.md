# Learny v3 Retrospective (RFC-003, Cycles A–F)

**Date**: 2026-07-20
**Scope**: The six cycles of [RFC-003](../rfc/0003-learny-v3-roadmap.md) that took Learny from a
deployed v2 product (v0.2.0) to a second-brain platform (v0.3.0): the operational and evaluation
maturity the flagship needed, scanned-PDF OCR, the research and design of the notes domain, and the
notes/second-brain loop itself — capture, retrieve, reinforce, export.

Each cycle ran as a `tlc-spec-driven` slice — specify → design → tasks → execute — with an independent
Verifier re-deriving coverage from the spec (author ≠ verifier), atomic per-task commits, and a small
reviewed PR. Decisions are recorded as ADRs; per-cycle decision logs live in `.specs/`. RFC-003 ordered
the arc deliberately: maturity tracks first (small, fully specified, hardening the platform the flagship
lands on), OCR next (independent extension of the Docling pipeline), then the flagship — researched and
designed before it was built, because the notes domain model was the one large unknown.

## What shipped, cycle by cycle

### Cycle A — Ops maturity (PR #28)

Hardened the deployed stack for a real host. A `backup` sidecar dumps PostgreSQL and syncs MinIO objects
off-VPS on a schedule with a retention policy and a **tested** restore path in `docs/ops/backups.md`;
self-hosted Netdata monitoring (`docs/ops/monitoring.md`) watches the compose stack with structured logs
still the correlation hook. Two ADR-0023 image-hygiene debts were paid: the runtime backend image no
longer installs the `dev` extra, and the PDF worker image was slimmed. This closed the backup and
monitoring half of the MVP TDD's long-open ops question ([ADR-0024](../adr/0024-backup-and-monitoring-stack.md)).

### Cycle B — Eval maturity (PR #35)

Turned the nightly evaluation from a deterministic placeholder into a real gate. Real-provider replay
snapshots (`text-embedding-3-large@1536` retrieval recall@k/MRR and Claude generation judge baselines)
were recorded via the existing `--record` paths and committed alongside the deterministic ones; judge
thresholds were calibrated from accumulated `eval-results` history and the aggregate gate armed in
`eval.yml` (`LEARNY_EVAL_GATE=1`), so a regression fails the nightly run while PRs stay offline and
deterministic. The calibration method is written down (`docs/ops/eval-calibration.md`) so a future model
swap re-derives thresholds instead of inheriting stale ones. This cycle shipped last of the six — after
the notes work — rather than in its roadmap position (see *What to improve*).

### Cycle C — Scanned-PDF OCR (PR #29)

Extended ingestion breadth without a new port. A scanned-PDF OCR path enables Docling's `do_ocr`
selectively behind the existing `DocumentParserPort`, with the OCR models handled like the baked layout
models and the `worker-pdf` memory/time budget re-verified. A table-driven localized normalization layer
(Portuguese-aware heading and front-matter patterns first, additive for other languages) landed in
`normalize_book`, and the golden fixtures gained a scanned sample
([ADR-0025](../adr/0025-selective-ocr-and-localized-normalization.md)).

### Cycle D — Second-brain research + design (PR #30)

The research gate for the flagship. A research fleet surveyed comparable products (RemNote, Readwise/Reader,
the Obsidian ecosystem, Recall), highlight anchoring below section granularity, notes data models, the
frontend editor stack, how notes should enter hybrid retrieval, note→quiz mapping, and export/portability —
with durable reports under `docs/research/`. The deliverable that unblocked the build was the notes-domain
ADR ([ADR-0026](../adr/0026-notes-and-second-brain-domain-model.md)), which locked the schema, retrieval,
edit-stability, and export decisions and confirmed the scopes of Cycles E and F. Per the RFC gate, those
build cycles did not start until it was accepted.

### Cycle E — Second-brain foundation: capture + organize (PR #31)

The capture layer. Notes (whole-Markdown) and highlights became user-owned records anchored to corpus
anchors with snapshot semantics so they survive re-ingestion (the quiz-item precedent), carrying tags and
note↔note / note↔section links. A reader selection creates a highlight or note; the app gained a notes
list and detail view, a backlinks panel in the reader, and edit/delete — the inert-but-organized corpus of
personal knowledge that Cycle F would put to work.

### Cycle F — Second-brain loop: retrieve + reinforce + export (this cycle)

Closed the loop and the release. Notes join hybrid retrieval as two additional RRF arms — a whole-note
`vector(1536)` embedding and a note `tsvector`, indexed in migration 0013, never entering `corpus_chunks`
— fused behind an `include_notes` toggle (default on for Q&A, off for teaching) and cited distinctly as
"your note". One action promotes a note to review cards through the existing quiz-generation port (the note
is the source, QC'd against its body); a note edit runs an async regenerate-and-match that rewrites matched
cards in place while leaving their scheduling and review-log rows byte-for-byte untouched — the cycle's core
invariant, guarded by a byte-equal sensor test — surfacing a "your note changed" badge with an explicit,
opt-in schedule reset. Making note cards possible required moving card ownership to a per-user column
(migration 0014) so a card can be source-less. Notes and highlights export as a deterministic, byte-identical
Obsidian vault (`GET /api/export/vault`). This cycle also refreshed the README, wrote this retrospective, and
versioned the product 0.3.0.

## What worked

- **Research-gating the flagship.** The one large unknown — the notes domain model — was designed before it
  was built. ADR-0026 locked schema, retrieval semantics, edit-stability, and export as a one-way projection,
  so the two build cycles executed against confirmed contracts instead of discovering the model mid-flight.
  The v2 retrospective's lesson (its flagship was researched up front) carried forward intact.
- **Ports made the flagship mechanical.** Notes joined retrieval as two more RRF arms inside the *same* SQL
  statement, and note-card generation reused the existing quiz-generation port — no new provider SDK, no
  orchestration framework, no domain rewrite. The second-brain loop is new product surface built almost
  entirely from existing seams.
- **Deterministic-first held through another provider-touching arc.** The new note-embedding and
  note-card-generation paths run on the network-free adapters by default; the backend suite grew from 1368 to
  1522 tests without the offline run ever requiring a key.
- **The edit-stability invariant was made testable, not asserted.** "Editing a note never disturbs a card's
  schedule" is the promise the reinforce feature lives or dies on; a sensor test compares scheduling and
  review-log rows byte-for-byte across an edit that rewrites the card text, so a regression reddens the suite
  rather than silently corrupting a learner's schedule.
- **Independent verification stayed honest.** Author ≠ verifier and the inject-a-fault sensor discipline
  continued to catch coverage gaps the implementer's own model would have missed.

## What to improve

- **Worker deaths and silent idling in the multi-worker ship model.** During the notes-loop build, one
  phase's worker died mid-task on a session interruption; a relaunched worker adopted the in-progress work
  tree cleanly with no collision, but the failure still cost a restart. Separately, workers occasionally went
  idle after finishing a phase without reporting, needing a manual nudge to continue or consolidate. Both are
  coordination gaps — the model would benefit from a heartbeat/timeout that detects a dead or idle worker
  instead of relying on a human noticing.
- **A test flake misdiagnosed as an infrastructure problem.** A "note ranks first" retrieval test was
  intermittently red, and the first hypothesis blamed HNSW index state bleeding across runs. The real cause
  was mundane: the `simple` full-text config doesn't stem the query term, so the note missed the lexical arm
  and tied the book's semantic hit on RRF score, with a random-UUID comparison deciding the order per run. The
  fix aligned the query tokens with the note body and made the index rebuild transactional; the assertions
  were left untouched. Reaching for the dramatic infrastructure explanation before the boring tie-break one
  cost debugging time — the sensor was working; the test was non-deterministic.
- **An ownership migration that the design should have anticipated.** Note cards are source-less by
  construction, but the quiz schema keyed authorization off a card's source. Making notes promotable forced a
  mid-cycle migration of card ownership to a per-card user column and a rewire of review authorization — a
  schema change (migration 0014) that would have been cheaper to fold into the notes design in Cycle D than to
  discover while wiring promotion.
- **The arc closed out of order and late.** Eval maturity (Cycle B) shipped last of the six, after the notes
  cycles, and the final cycle closed interleaved with later work. The maturity tracks were sequenced first
  precisely to harden the platform before the flagship landed; deferring one past the flagship is defensible
  but blurs the "harden first" intent the ordering was chosen for.

## Forward look

Post-v3 candidates, most recorded as deferred by ADR-0026 rather than open questions:

- Paragraph-level note chunking for retrieval — ADR-0026 embeds whole notes first; chunking is the recorded
  upgrade once whole-note relevance stops being enough.
- Fuzzy re-anchoring, a graph UI, and a richer block/outliner editor — deferred beyond v3 by ADR-0026 scope.
- Vault import and round-trip sync — export is deliberately a one-way projection, never a sync (ADR-0026 d6).
- Notes in *teaching* retrieval defaulting on — off for teaching until the value is proven.
- Multi-provider / bring-your-own-key generation and embeddings, a dedicated vector database or reranker if
  PostgreSQL hybrid search stops scaling, and a publicly hosted multi-tenant instance (all still out of scope,
  no decision recorded).
